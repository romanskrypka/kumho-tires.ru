# Развёртывание на хостинге с FTP

Инструкция для выкладки проекта на обычный shared-хостинг, где доступ только по FTP и нет симлинков.

## Что нужно на хостинге

- **PHP 8.2+** (лучше 8.5, как в проекте), расширения: json, mbstring
- **Apache** с `mod_rewrite` и `AllowOverride All` (или аналог в панели)
- Возможность указать **корень сайта (Document Root)** на подпапку (например `public`)

Если корень сайта нельзя сменить (всегда только `public_html`), придётся выносить только содержимое `public/` в корень и поднимать уровень выше `config/`, `src/`, `templates/`, `vendor/` — такой вариант здесь не описан, лучше уточнить у хостера смену Document Root.

## Шаг 1. Сборка и подготовка public (локально)

На своём компьютере в корне проекта:

```bash
# Полная продакшн-сборка (линты, тесты, CSS/JS)
npm run build
```

Затем заменить симлинки на реальные копии (для FTP симлинки не подойдут):

```bash
npm run setup:public-copy
```

В каталоге `public/` появятся папки `assets/` и `data/` (копии) и файл `robots.txt`. Дальше загружаем именно такую структуру.

## Шаг 2. Что загружать по FTP

Загружать нужно **весь проект** в каталог, который будет корнем приложения (например `domains/site.com/` или `www/`), так, чтобы на сервере получилось:

```
корень_сайта/
├── public/          ← сюда позже укажете Document Root
│   ├── index.php
│   ├── .htaccess
│   ├── llms.txt
│   ├── llms-full.txt
│   ├── robots.txt   (копия из корня)
│   ├── assets/      (копия: css/build, js/build, fonts, img…)
│   └── data/        (копия: json, img…)
├── config/
├── src/
├── templates/
├── vendor/          (composer install — см. ниже)
├── cache/           (пустая папка, права на запись)
├── logs/            (пустая папка, права на запись)
├── data/            (можно не заливать, если уже есть public/data)
├── assets/          (можно не заливать, если уже есть public/assets)
├── .env
├── .env.example
├── composer.json
└── composer.lock
```

Важно:

- В `public/` должны лежать **реальные** каталоги `assets` и `data` (после `npm run setup:public-copy`), а не симлинки.
- Папки `cache/` и `logs/` создайте на сервере, если их нет, и выставьте права на запись (обычно 755 или 775).

## Шаг 3. Document Root (корень сайта)

В панели хостинга (cPanel, ISPmanager, «Управление доменом» и т.п.) укажите в качестве **корня сайта (Document Root)** каталог **`public`** (или `public/`), а не корень проекта.

Тогда веб-сервер будет отдавать только то, что в `public/`, а файлы `.env`, `config/`, `src/` останутся вне веб-доступа.

## Шаг 4. PHP-зависимости на сервере

Если на хостинге есть SSH:

```bash
cd /путь/к/корню_сайта
composer install --no-dev --optimize-autoloader
```

Если только FTP — установите зависимости локально и залейте папку `vendor/` по FTP (в корень проекта, рядом с `config/`, `src/`, `public/`).

## Шаг 5. Файл .env на сервере

Создайте на сервере файл `.env` в **корне проекта** (рядом с `composer.json`), не внутри `public/`. Скопируйте содержимое из `.env.example` и задайте:

```env
APP_ENV=production
APP_DEBUG=0
APP_BASE_URL=https://ваш-домен.ru/
APP_DEFAULT_LANG=ru
```

Файл `.env` не должен быть доступен по URL (поэтому Document Root должен быть именно `public/`).

## Шаг 6. Проверка после выкладки

- Откройте главную и несколько страниц — должны отдаваться 200.
- Проверьте, что подгружаются CSS и JS: DevTools → вкладка Network.
- Проверьте картинки: `/data/img/...`, `/assets/...`.
- Несуществующий URL должен отдавать 404.

## Обновление сайта (релиз)

1. Локально: `git pull`, затем `npm run build` и `npm run setup:public-copy`.
2. По FTP залейте обновлённые файлы:
   - содержимое `public/` (в т.ч. `public/assets`, `public/data`),
   - при изменении PHP/конфигов — `config/`, `src/`, `templates/`, при необходимости заново залейте `vendor/` или выполните на сервере `composer install`.
3. При смене только статики достаточно обновить `public/`.

## Краткий чеклист

- [ ] Локально: `npm run build`, затем `npm run setup:public-copy`
- [ ] На сервер залит весь проект, в `public/` — реальные папки `assets` и `data`
- [ ] Document Root в панели хостинга указывает на каталог `public`
- [ ] В корне проекта (не в public) создан и заполнен `.env`
- [ ] Папки `cache/` и `logs/` существуют и доступны для записи
- [ ] При необходимости залита папка `vendor/` или выполнен `composer install` на сервере
