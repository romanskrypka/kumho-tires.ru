1. Проблема

Сейчас платформа существует в одном репозитории с единственным набором данных. При переходе на нового заказчика возникает дилемма:

Удалить старый контент

Теряем рабочие шаблоны, JS, CSS — доказанные паттерны (каталог с фильтрацией, карточки товаров, дилерская сеть).

Оставить старый контент

Мусор в проекте нового заказчика: JSON-данные, изображения шин, упоминания Kumho.

Корневая причина: нет формального разделения между платформой (ядро + библиотека компонентов) и проектом (данные + конфигурация конкретного заказчика).

2. Целевая модель

2.1 Три слоя системы

Слой 3: Данные проекта
JSON, изображения, .env, project.php — уникальны для каждого заказчика. Живут в ветке проекта.
Слой 2: Библиотека компонентов
Twig-шаблоны, JS, CSS секций и компонентов — общие для всех проектов, накапливаются. Живут в main.
Слой 1: Ядро платформы
PHP (src/), config/settings.php, base.twig, page.twig, Webpack, PostCSS, scaffold. Не меняется между проектами.

2.2 Принцип

Проект = ветка от платформы + свои данные.
Каждый заказчик — это git-ветка, отходящая от main. Ветка содержит свой project.php, data/json/, data/img/, .env, variables.css. Всё остальное — наследуется от платформы и обновляется через git merge main.

3. Классификация файлов

3.2 Проект (Слой 3) — менять для каждого заказчика

КатегорияДействие
config/project.phpПереписать: route_map, collections, sitemap_pages
.envПереписать: APP_BASE_URL, YANDEX_METRIC_ID, API-ключи

data/json/{lang}/seo/*.jsonПереписать
data/json/{lang}/{collection}/*.jsonУдалить старые коллекции, создать новые
data/json/global.jsonОбновить: навигация, контакты, логотипы
data/img/Удалить старые изображения, добавить новые

Ключевое правило

Шаблоны, JS и CSS — это библиотека платформы, а не контент проекта. Неиспользуемый шаблон (без JSON) — это неактивный компонент, а не мусор. Удалять шаблоны при смене проекта — всё равно что удалять неиспользуемые классы из Bootstrap.

4. Git-стратегия

4.1 Структура веток

main (платформа — ядро + ВСЕ шаблоны)
  │
  ├── project/kumho ← рабочий deployment Kumho Tires
  │    project.php (tires, news), data/json/ (Kumho), variables.css (красный)
  │
  ├── project/retail-logistic ← Ритейл Логистик
  │    project.php (demo-tools), data/json/ (РЛ), variables.css (оранжевый)
  │
  └── project/client-3 ← будущий заказчик

4.2 Обновление платформы в проекте

# Находясь в ветке проекта:
git merge main

# Конфликты возможны только в:
# - config/project.php
# - data/json/global.json
# - assets/css/base/variables.css
# - main.css / main.js (новые импорты)

4.3 Создание нового проекта

# 1. Ветка от main (со ВСЕМИ шаблонами)
git checkout main
git checkout -b project/new-client

# 2. Удалить ТОЛЬКО данные (JSON + изображения)
rm -rf data/json/ru/tires/ data/json/ru/news/
rm -rf data/img/tires/ data/img/news/ data/img/actions/
rm -f data/json/ru/pages/{tires-list,tires,news,buy,warranty,dealers}.json

# 3. Настроить проект
# Обновить: project.php, global.json, variables.css, .env

# 4. Наполнить контентом
npm run create-page -- demo-tools

6. План действий

0
Восстановление
1-2 часа
Восстановить из git удалённые шаблоны (9 Twig), JS (2 файла), CSS (11 файлов) и импорты в main.js / main.css. JSON-данные Kumho остаются удалёнными — это правильно.
Результат: полная библиотека компонентов + данные РЛ.
1
Формализация структуры
2-4 часа
Создать ветку project/kumho от коммита до зачистки. Создать ветку project/retail-logistic от main. Задокументировать ветвление в README.
Результат: чёткое разделение платформы и проектов на уровне git.
2
Playbook тиражирования
2-3 часа
Формализованная инструкция: чеклист входных данных, пошаговое создание ветки проекта, что удалять / что не трогать / что настраивать, чеклист верификации.
Результат: любой разработчик может создать проект для нового заказчика.
3
Переименование шаблонов (опционально)
1-2 часа
card-tire → card-product, tires → catalog, dealers → partners-list.
Альтернатива: оставить текущие имена + задокументировать универсальное назначение. Проще и менее рискованно.
4
Завершение проекта РЛ
текущая работа
Продолжить наполнение Ритейл Логистик в ветке project/retail-logistic. Главная, контакты, demo-tools, policy, agree, 404 — структура готова, ожидаем реальные данные от заказчика.
