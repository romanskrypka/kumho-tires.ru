### Метафора

- **PHP = Лицо** — что видит пользователь. Красивое, быстрое, SEO-оптимизированное
- **n8n = Нервы** — как передаётся сигнал. Связывает компоненты, реагирует на события
- **Django = Мозг** — где принимаются решения. AI, бизнес-логика, данные

## Карта ответственности

### Кто за что отвечает

| Компонент | Что ДЕЛАЕТ | Что НЕ делает | Сильные стороны | Слабые стороны |
|-----------|-----------|--------------|----------------|----------------|
| **PHP** | UI, контент, SEO, сессии, формы, API-прокси | AI/ML обработка, тяжёлые вычисления | Быстрый рендеринг, огромная экосистема, PHP-FPM + nginx = отличная производительность | AI-логика — чужая зона, масштабирование требует усилий |
| **n8n** | Оркестрация workflow, триггеры (cron/webhook), маршрутизация вызовов, интеграции с внешними сервисами | Обработка данных, парсинг файлов, ML, тяжёлая бизнес-логика | 400+ готовых интеграций, визуальный редактор, быстрое прототипирование | Масштаб ограничен, один инстанс = потолок, Queue Mode с Redis улучшает ситуацию |
| **Django** | Бизнес-логика, AI-агенты, ML-модели, REST API, Celery-задачи, персонализация, данные | UI рендеринг, SEO | Python AI/ML экосистема (LangChain, PyTorch, Transformers), async views, Celery + Redis для очередей, горизонтальное масштабирование | Overhead для простых задач, требует знания Python-стека |

```

  ✅ Отобразить результат (после callback)

  ❌ НЕ обрабатывает изображение

n8n:
  ✅ Принять webhook от PHP
  ✅ Определить тип задачи → маршрутизировать
  ✅ Вызвать Django Agents API
  ✅ Получить результат → callback в PHP
  ❌ НЕ парсит изображение
  ❌ НЕ вызывает Vision API напрямую
  ❌ НЕ хранит результаты

Django (Agents):
  ✅ Принять задачу через REST API
  ✅ Вызвать Vision LLM (OpenAI, Claude)
  ✅ Распарсить ответ, извлечь данные
  ✅ Сохранить в БД, вернуть результат
  ✅ Обновить историю пользователя
```

## Оценка по 8 измерениям

### 1. Переиспользуемость — ✅✅ Высокая

**Кто даёт:** Django apps + n8n шаблоны + PHP Composer

| Компонент | Механизм переиспользования | Пример |
|-----------|--------------------------|--------|
| Django | Django apps — модульные по природе | `agents/` — одно приложение используется во всех сервисах |
| n8n | Workflow сохраняются как JSON-шаблоны | Workflow «уведомление» вызывается из 10 разных мест |
| PHP | Composer-пакеты, DI-контейнер, Twig-компоненты | `DataLoaderService` используется для всех коллекций |

**Архитектурное решение:** `BaseAgent` (Python ABC) + `loadEntity` (PHP universal) — контракты фиксированы, реализации индивидуальны.

**Антипаттерн:** Копирование кода между Django-сервисами вместо общей библиотеки `django-common/`.

---

### 2. Тиражируемость — ✅✅ Отличная

**Кто даёт:** Docker + вся связка

```yaml
# docker-compose.yml — вся система одной командой
services:
  php:
    image: php:8.5-fpm
    volumes:
      - ./config/project.php:/app/config/project.php
      - ./data/json:/app/data/json
      - ./assets/css/base/variables.css:/app/assets/css/base/variables.css

  n8n:
    image: n8nio/n8n
    environment:
      - N8N_ENCRYPTION_KEY=${N8N_ENCRYPTION_KEY}

  django-core:
    build: ./django-services/core-api
    environment:
      - DATABASE_URL=${DATABASE_URL}

  django-agents:
    build: ./django-services/agents-service
    environment:
      - OPENAI_API_KEY=${OPENAI_API_KEY}

  postgres:
    image: pgvector/pgvector:pg16

  redis:
    image: redis:7-alpine
```

**Тиражирование = Docker-образ.** Новый deployment:
1. Клонировать `deployments/template/`
2. Заполнить `.env`
3. Адаптировать `project.php`, `data/json/`, `variables.css`
4. `docker-compose up -d`

**Антипаттерн:** Ручная настройка без Docker — каждый deployment = уникальный набор проблем.

---

### 3. Автономность — ✅✅ Высокая

**Кто даёт:** n8n триггеры + Django агенты

```
n8n:
  Cron-триггеры → запускают агентов по расписанию
  Webhook-триггеры → реакция на события
  Условная логика → маршрутизация без участия человека

Django Agents:
  BaseAgent.run() → validate → process → return
  Автономный pipeline без человека

n8n + Django вместе:
  Агент завершил задачу →
    если ошибка → n8n перезапускает (max 3 раза)
    если успех → n8n передаёт результат в PHP
    если нужен человек → n8n уведомляет (Telegram, email)
```

**Архитектурное решение:** Спектр автономности из [tz_identity.md](tz_identity.md) — `confidence_threshold` и `autonomy_level` в `project.php` определяют, когда агент решает сам, а когда запрашивает подтверждение.

**Антипаттерн:** Агент всегда автономен без учёта роли пользователя (confidence 50% → всё равно автоматически).

### 4. Масштабирование — ⚠️ Требует внимания

**Главное узкое место: n8n.**

| Компонент | Масштабируемость | Механизм | Потолок |
|-----------|:----------------:|----------|---------|
| PHP | ✅ Хорошая | PHP-FPM + nginx, горизонтально | Тысячи RPS |
| Django | ✅ Хорошая | Gunicorn + несколько workers, Celery, async views | Тысячи RPS + фоновые задачи |
| n8n | ⚠️ Ограниченная | Один инстанс, Queue Mode с Redis улучшает | Сотни workflow/мин |

**Решение: правильное распределение нагрузки:**

```
ПРАВИЛЬНО:                              НЕПРАВИЛЬНО:

PHP ──[webhook]──> n8n                   PHP ──[webhook]──> n8n
                    │                                        │
              [маршрутизация]                          [обработка данных]
                    │                                   [парсинг файлов]
                    ▼                                   [вызов AI API]
               Django Celery                            [формирование ответа]
                    │                                        │
              [тяжёлая работа]                         n8n = монолит
              [AI, ML, данные]                         = bottleneck
                    │                                   = точка отказа
                    ▼
               n8n [callback]
                    │
                    ▼
                  PHP
```

**Конкретные правила масштабирования:**

| Правило | Описание | Зачем |
|---------|---------|-------|
| n8n workflow: max 10-15 нод | Если больше — часть логики переносится в Django | n8n не для сложной логики |
| n8n НЕ парсит файлы | Обработка изображений, PDF, CSV — только Django | CPU-интенсивные задачи убивают n8n |
| n8n НЕ делает ML/AI | Вызовы LLM, Vision API — только Django | Длинные запросы блокируют n8n |
| n8n Queue Mode + Redis | Self-hosted n8n с очередями | Надёжность и параллельность |
| > 1000 вызовов/час → Celery | Высоконагруженные workflow перемещаются в Django Celery | n8n не для высокой нагрузки |
| Health checks на каждом сервисе | Docker health checks + автоматический рестарт | Упавший сервис = автовосстановление |

**Архитектура масштабирования:**

```
                    Load Balancer
                         │
              ┌──────────┼──────────┐
              │          │          │
           PHP #1     PHP #2     PHP #3
              │          │          │
              └──────────┼──────────┘
                         │
                    n8n (1 инстанс)
                    Queue Mode + Redis
                         │
              ┌──────────┼──────────┐
              │          │          │
         Django #1  Django #2  Django #3
         + Celery   + Celery   + Celery
              │          │          │
              └──────────┼──────────┘
                         │
                    PostgreSQL
                    + Redis
                    + pgvector
```

PHP и Django масштабируются горизонтально (replicas). n8n — вертикально (один мощный инстанс + Queue Mode). Это основной компромисс стека.

### 5. Гибкость — ✅✅ Очень высокая

**Кто даёт:** Вся связка — три разных языка и парадигмы.

| Потребность | Решение | Компонент |
|------------|---------|-----------|
| Быстрый UI | Twig + JSON-контент | PHP |
| Интеграция с CRM | Готовый коннектор | n8n |
| AI-агент | LangChain / Anthropic SDK | Django |
| Новый агент | Наследник BaseAgent | Django |
| Новый триггер | Webhook/Cron workflow | n8n |
| Новая страница | JSON в `data/json/` | PHP |
| Замена оркестратора | n8n → Temporal/Airflow | DI + интерфейс `AutomationGatewayInterface` |

**Ключевое:** Замена любого компонента не ломает остальные — взаимодействие через контракты (REST API, Webhooks).

**Антипаттерн:** Прямые вызовы между компонентами без интерфейса-контракта (PHP напрямую вызывает внутренний эндпоинт Django, не через `DjangoApiClient`).

### 6. Универсальность — ✅✅ Одна из лучших связок

**Кто даёт:** n8n (главный вклад) + Django (Python-экосистема)

```
n8n: 400+ готовых интеграций
├── CRM: Salesforce, HubSpot, Bitrix24
├── Мессенджеры: Telegram, Slack, WhatsApp
├── БД: PostgreSQL, MongoDB, MySQL
├── AI: OpenAI, Anthropic, Ollama, HuggingFace
├── Email: Gmail, SMTP, Outlook
├── Файлы: Google Drive, S3, Dropbox
├── Платежи: Stripe
├── + любой HTTP/webhook
└── + MCP-нода (подключение MCP Server)

Django: любая Python-библиотека
├── AI/ML: LangChain, Transformers, PyTorch
├── Vision: Pillow, OpenCV
├── NLP: spaCy, NLTK
├── Данные: pandas, numpy
└── + любой pip-пакет
```

**PHP добавляет:** Любой веб-стандарт (OAuth 2.0, REST, GraphQL, WebSocket через Ratchet).

**Антипаттерн:** Реализовывать интеграцию с внешним сервисом в PHP-коде, когда в n8n есть готовый коннектор.

### 7. Индивидуальность — ✅ Да, но создаёшь ТЫ

**Кто даёт:** Архитектура + решения (не инструменты сами по себе).

Индивидуальность — единственное измерение, которое **не дают инструменты автоматически**. Конкуренты используют те же PHP, n8n, Django. Индивидуальность = ЧТО ты с ними делаешь.

| Уровень индивидуальности | Механизм | Компонент |
|-------------------------|----------|-----------|
| Уникальные AI-агенты | `extensions/custom-agents/` | Django |
| Уникальные бизнес-процессы | `extensions/custom-workflows/` | n8n |
| Уникальный UI/UX | `data/json/` + `variables.css` + `templates/override/` | PHP |
| Уникальные MCP Prompts | Доменные инструкции для AI | Django MCP Server |
| Уникальные данные | Справочники, классификаторы | Django Core + JSON |

**Связь с [tz_identity.md](tz_identity.md):** Индивидуальность deployment'а = проектный слой (`deployments/{client}/` + `extensions/`). Ядро (`core/`) — одинаковое, характер — уникальный.

### 8. Персональность — ✅✅✅ Главный козырь связки

**Кто даёт:** Django + Vector DB + Redis. Это самое сильное измерение стека.

Именно здесь связка PHP + n8n + Django **сильнее многих других решений**, потому что Django + Python = прямой доступ к инструментам персонализации (LangChain memory, pgvector, embeddings).

**Архитектура персонализации:**

```
╔═════════════════════════════════════════════════════╗
║            СЛОЙ ПАМЯТИ ПОЛЬЗОВАТЕЛЯ                  ║
╠════════════════════╦════════════════════════════════╣
║  Краткосрочная     ║  Долгосрочная                   ║
║  Redis             ║  PostgreSQL + pgvector           ║
║                    ║                                  ║
║  • Текущая сессия  ║  • История задач                 ║
║  • Контекст        ║  • Предпочтения                 ║
║    диалога         ║  • Решения (принятые/отклонённые)║
║  • Последний       ║  • Эмбеддинги взаимодействий    ║
║    запрос          ║  • Семантический поиск            ║
╚════════════════════╩════════════════════════════════╝
                     │
╔════════════════════▼════════════════════════════════╗
║              Django Agent                            ║
║                                                      ║
║  def get_personalized_response(user, query):         ║
║      profile = user.get_profile()        # роль      ║
║      memory = user.get_memory()          # история   ║
║      context = vector_search(query, user) # семантика║
║      return agent.run(                               ║
║          query=query,                                ║
║          profile=profile,                            ║
║          memory=memory,                              ║
║          context=context,                            ║
║      )                                               ║
╚════════════════════╦════════════════════════════════╝
                     │
╔════════════════════▼════════════════════════════════╗
║                    n8n                                ║
║                                                      ║
║  Триггер: пользователь загрузил этикетку             ║
║  → Получить профиль пользователя                     ║
║  → Получить историю (последние 10 задач)             ║
║  → Вызвать Django Agent с контекстом                 ║
║  → Обновить память пользователя                      ║
║  → Callback → PHP (персонализированный ответ)        ║
╚══════════════════════════════════════════════════════╝
```

**Слоистая персонализация (из [tz_identity.md](tz_identity.md)):**

| Слой | Хранилище | Стоимость | Этап |
|------|-----------|:---------:|:----:|
| 1. Роль | `project.php` | ~0 | 2-3 |
| 2. Настройки | Django Core (UserProfile) | ~1 KB/user | 3 |
| 3. История | Django Core (TaskHistory) | ~100 KB/мес/user | 4-5 |
| 4. Семантическая память | pgvector (embeddings) | ~10 MB/user | 5+ |

| Redis | 3 | JWT-сессии + кэш справочников |
| Celery + Redis | 4 | Очереди для AI-задач |
| pgvector | 5 | Семантическая память для персонализации |

## Критический риск: n8n как bottleneck

### Проблема

n8n спроектирован как **low-code orchestrator**, не как **high-throughput message broker**. При неправильном использовании n8n становится монолитом:

```
❌ АНТИПАТТЕРН: n8n = центр вселенной

Всё → n8n → n8n → n8n → n8n → результат

n8n обрабатывает данные, парсит файлы,
вызывает API, формирует ответ, сохраняет результат.

Результат: 50+ нод в workflow, неуправляемо,
не масштабируется, одна ошибка = всё падает.
```

### Решение: n8n = дирижёр, НЕ исполнитель

```
✅ ПРАВИЛЬНАЯ МОДЕЛЬ:

PHP (запрос) → n8n (маршрутизация) → Django (исполнение)
                                           │
                                      [тяжёлая работа]
                                      [AI, ML, данные]
                                           │
                                      n8n (callback)
                                           │
                                      PHP (ответ)

n8n НЕ обрабатывает данные.
n8n МАРШРУТИЗИРУЕТ вызовы.
Django Celery — для тяжёлых задач.
```

### Правила использования n8n

| # | Правило | Обоснование |
|---|---------|-------------|
| 1 | Max 10-15 нод в workflow | Если больше — часть логики в Django service |
| 2 | n8n НЕ парсит файлы | Изображения, PDF, CSV → Django |
| 3 | n8n НЕ делает ML/AI | Vision API, LLM, embeddings → Django |
| 4 | n8n НЕ хранит данные | Результаты → Django Core (PostgreSQL) |
| 5 | Queue Mode + Redis | Self-hosted n8n с очередями для надёжности |
| 6 | > 1000 вызовов/час → Celery | Высоконагруженные задачи → Django Celery |
| 7 | Retry policy: max 3 раза | 5xx от Django → n8n перезапускает (с backoff) |
| 8 | Timeout: 30 сек на webhook | n8n не ждёт долго, Django обрабатывает асинхронно |

### Когда n8n, когда Celery

| Критерий | n8n | Django Celery |
|---------|-----|---------------|
| Частота | < 1000/час | > 1000/час |
| Длительность | < 30 сек | > 30 сек |
| Сложность | Маршрутизация, условия, интеграции | Обработка данных, AI, ML |
| Визуализация | Нужна (бизнес-процессы) | Не нужна (фоновые задачи) |
| Retry | Простой (3 раза) | Сложный (exponential backoff, dead letter queue) |
