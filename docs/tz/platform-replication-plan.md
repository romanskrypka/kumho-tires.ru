# План развития и тиражирования iSmart Platform

> **Версия:** 1.0
> **Дата:** 2026-03-10
> **Статус:** Проект для обсуждения
> **Контекст:** Платформа содержит content-agnostic PHP-ядро (v1.1.0), рабочий deployment Kumho Tires и начатый переход на «Ритейл Логистик». Необходимо формализовать процесс тиражирования, чтобы работа над новым проектом не разрушала reference-реализацию.

---

## 1. Проблема

Сейчас платформа существует в одном репозитории с единственным набором данных. При переходе на нового заказчика возникает дилемма:

- **Удалить старый контент** → теряем рабочие шаблоны, JS, CSS — доказанные паттерны (каталог с фильтрацией, карточки, дилеры)
- **Оставить старый контент** → мусор в проекте нового заказчика, путаница

Корневая причина: **нет формального разделения** между платформой (ядро + библиотека компонентов) и проектом (данные + конфигурация конкретного заказчика).

## 2. Целевая модель

### 2.1 Три слоя системы

```
┌─────────────────────────────────────────────────┐
│  СЛОЙ 3: ДАННЫЕ ПРОЕКТА                         │
│  JSON, изображения, .env, project.php            │
│  Уникальны для каждого заказчика                 │
│  Живут: ветка проекта / deployment               │
├─────────────────────────────────────────────────┤
│  СЛОЙ 2: БИБЛИОТЕКА КОМПОНЕНТОВ                  │
│  Twig-шаблоны, JS, CSS секций и компонентов      │
│  Общие для всех проектов, накапливаются           │
│  Живут: ветка platform (main)                    │
├─────────────────────────────────────────────────┤
│  СЛОЙ 1: ЯДРО ПЛАТФОРМЫ                          │
│  PHP (src/), config/settings.php, base.twig,     │
│  page.twig, Webpack, PostCSS, scaffold-генераторы│
│  Не меняется между проектами                     │
│  Живут: ветка platform (main)                    │
└─────────────────────────────────────────────────┘
```

### 2.2 Принцип: проект = форк платформы + свои данные

Каждый новый заказчик — это **ветка** (или форк), отходящая от `main` (платформы). Ветка проекта содержит:
- Свой `config/project.php`
- Свой `data/json/` (страницы, SEO, коллекции)
- Свои `data/img/` (изображения, логотипы, фавиконы)
- Свой `.env`
- Свои переменные в `variables.css` (цвета, шрифты)

Всё остальное — **наследуется от платформы** и обновляется через `git merge` из `main`.

### 3.2 Что относится к проекту (Слой 3)

| Категория | Файлы | При тиражировании |
|-----------|-------|-------------------|
| **Конфигурация проекта** | `config/project.php` | Переписать под заказчика |
| **Переменные окружения** | `.env` | Переписать |
| **Контент страниц** | `data/json/{lang}/pages/*.json` | Переписать |
| **SEO** | `data/json/{lang}/seo/*.json` | Переписать |
| **Коллекции** | `data/json/{lang}/{collection}/*.json` | Удалить старые, создать новые |
| **Глобальные данные** | `data/json/global.json` | Переписать (навигация, контакты, логотипы) |
| **Изображения контента** | `data/img/tires/`, `data/img/news/`, `data/img/actions/`, `data/img/frames/` | Удалить старые, добавить новые |
| **Логотипы** | `data/img/ui/logos/*` | Заменить |
| **Фавиконы** | `data/img/favicons/*` | Заменить |
| **OG-изображение** | `data/img/seo/*` | Заменить |
| **Фирменные цвета** | `assets/css/base/variables.css` (частично) | Обновить цвета |
| **Метафайлы** | `public/robots.txt`, `public/llms.txt` | Обновить |

### 3.3 Ключевое правило

> **Шаблоны, JS и CSS — это библиотека платформы, а не контент проекта.**
> Неиспользуемый шаблон (без JSON, который на него ссылается) — это неактивный компонент библиотеки, а не мусор.
> Удалять шаблоны при смене проекта — всё равно что удалять неиспользуемые классы из Bootstrap.

## 4. Git-стратегия

### 4.1 Структура веток

```
main (платформа)
  │
  ├── project/kumho          ← ветка проекта Kumho Tires
  │     Отличия от main:
  │     - config/project.php (tires, news collections)
  │     - data/json/ (контент Kumho)
  │     - data/img/ (изображения Kumho)
  │     - .env (Kumho settings)
  │     - assets/css/base/variables.css (цвета Kumho)
  │
  ├── project/retail-logistic ← ветка проекта Ритейл Логистик
  │     Отличия от main:
  │     - config/project.php (demo-tools, без коллекций)
  │     - data/json/ (контент РЛ)
  │     - data/img/ (изображения РЛ)
  │     - .env (РЛ settings)
  │     - assets/css/base/variables.css (цвета Vintegra)
  │
  └── project/client-3       ← будущий заказчик
```

### 4.2 Как обновить платформу в проекте

```bash
# Находясь в ветке проекта:
git merge main

# Конфликты возможны только в:
# - config/project.php (специфика проекта — resolve manually)
# - data/json/global.json (навигация, контакты — resolve manually)
# - assets/css/base/variables.css (цвета — resolve manually)
# - assets/css/main.css и assets/js/main.js (новые импорты — accept both)
```

### 4.3 Как создать новый проект

```bash
# 1. Создать ветку от main (платформа со ВСЕМИ шаблонами)
git checkout main
git checkout -b project/new-client

# 2. Удалить ТОЛЬКО данные предыдущего проекта (JSON + изображения)
rm -rf data/json/ru/tires/ data/json/ru/news/
rm -rf data/img/tires/ data/img/news/ data/img/actions/ data/img/frames/
rm -f data/json/ru/pages/{tires-list,tires,news,buy,warranty,dealers}.json
rm -f data/json/ru/seo/{tires-list,news,buy,warranty}.json

# 3. Настроить проект через scaffold
npm run create-page -- demo-tools
# Обновить config/project.php, global.json, variables.css, .env

# 4. Наполнить контентом
# Редактировать data/json/ru/pages/*.json, seo/*.json
```

## 5. Что происходит с неиспользуемыми шаблонами

### 5.1 Twig-шаблоны без JSON

Шаблон `sections/tires.twig` существует, но ни один JSON не содержит `"name": "tires"` в `sections[]`.

**Результат:** шаблон не вызывается. `page.twig` итерирует только те секции, которые перечислены в JSON. Мёртвый файл = нулевой эффект на рендеринг.

### 5.2 CSS/JS без шаблона

`tires.css` и `tires.js` импортированы в `main.css` / `main.js`, но DOM-элементы с соответствующими классами и селекторами отсутствуют.

**CSS:** мёртвые правила — не применяются, но занимают место в бандле. При сборке cssnano минифицирует, но не удаляет неиспользуемые правила (для этого нужен PurgeCSS).

**JS:** секции используют паттерн `document.querySelector('.tires')`. Если элемента нет — код не выполняется. Мёртвый код инициализации = пара десятков байт в бандле.

### 5.3 Оценка «стоимости» неиспользуемых компонентов

| Метрика | Значение | Влияние |
|---------|----------|---------|
| Размер CSS 11 удалённых файлов | ~15-25 KB (до минификации) | ~3-5 KB gzip — пренебрежимо |
| Размер JS 2 удалённых файлов | ~5-10 KB | ~1-2 KB gzip — пренебрежимо |
| Кол-во файлов в проекте | +20 файлов | Не влияет на пользователя |
| Время сборки | +50-100ms | Пренебрежимо |
| Риск поломки при удалении | Средний | Забытый импорт = ошибка сборки |

**Вывод:** оставлять дешевле и безопаснее, чем удалять.

## 6. План действий

### Фаза 0: Восстановление (1-2 часа)

> Привести репозиторий к правильному состоянию

| # | Действие | Файлы |
|---|----------|-------|
| 1 | Восстановить из git удалённые шаблоны Twig | `tire.twig`, `news.twig` (pages), `tires.twig`, `news.twig`, `dealers.twig` (sections), `card-tire.twig`, `card-news.twig`, `card-dealer.twig`, `filter.twig` (components) |
| 2 | Восстановить из git удалённые JS | `tires.js`, `dealers.js` |
| 3 | Восстановить из git удалённые CSS | 11 файлов (pages/tires, pages/news, sections/tires, sections/news, sections/dealers, components/filter, card-tire, card-news, card-dealer) |
| 4 | Восстановить импорты в `main.js` | Добавить обратно `import './sections/tires.js'`, `import './sections/dealers.js'` |
| 5 | Восстановить импорты в `main.css` | Добавить обратно все 11 `@import` |
| 6 | Проверить сборку | `npm run build:dev` — должна пройти без ошибок |

**Результат:** платформа содержит полную библиотеку компонентов. JSON-данные Kumho остаются удалёнными (это правильно). Шаблоны лежат неактивными.

### Фаза 1: Формализация структуры (2-4 часа)

> Разделить платформу и проект на уровне git

| # | Действие | Описание |
|---|----------|----------|
| 1 | Коммит текущего состояния в `main` | Платформа + библиотека компонентов + данные РЛ |
| 2 | Создать ветку `project/kumho` от коммита до зачистки | `git branch project/kumho <commit-hash>` — для сохранения рабочего Kumho |
| 3 | Создать ветку `project/retail-logistic` от `main` | Рабочая ветка для РЛ |
| 4 | Документировать ветвление в README | Описать роль каждой ветки |

### Фаза 2: Playbook тиражирования (2-3 часа)

> Создать формализованную инструкцию «как создать проект для нового заказчика»

**Содержание playbook'а:**

1. **Чеклист входных данных от заказчика** — логотип, цвета, контакты, контент, юридические реквизиты
2. **Пошаговая инструкция создания ветки проекта** — от `git checkout -b` до `npm run build:dev`
3. **Что удалять** (только JSON-данные и изображения предыдущего проекта)
4. **Что НЕ удалять** (шаблоны, JS, CSS — никогда)
5. **Что настраивать** (`project.php`, `global.json`, `variables.css`, `.env`)
6. **Как добавлять новые секции** (`npm run create-section`)
7. **Как добавлять коллекции** (`npm run create-collection`)
8. **Чеклист верификации** (JSON-валидация, сборка, smoke-тесты, визуальная проверка)

### Фаза 3: Генеалогия шаблонов (1-2 часа)

> Переименовать контент-специфичные шаблоны в универсальные

| Текущее имя | Предлагаемое имя | Причина |
|-------------|-------------------|---------|
| `card-tire.twig` | `card-product.twig` | Универсальная карточка товара |
| `card-news.twig` | `card-article.twig` | Универсальная карточка статьи |
| `card-dealer.twig` | `card-partner.twig` | Универсальная карточка партнёра |
| `tires.twig` (section) | `catalog.twig` | Каталог с фильтрацией |
| `news.twig` (section) | `articles.twig` | Лента статей |
| `dealers.twig` (section) | `partners-list.twig` | Список партнёров/точек |
| `tire.twig` (page) | `product.twig` | Детальная страница товара |
| `news.twig` (page) | `article.twig` | Детальная страница статьи |

**Важно:** переименование должно сопровождаться обновлением:
- Ссылок в JSON (`"name": "tires"` → `"name": "catalog"`)
- Импортов в `main.css` / `main.js`
- Имён CSS-классов (`.tires` → `.catalog`)
- Конфигурации в `project.php` (template: `pages/product.twig`)

**Альтернатива:** оставить текущие имена, но задокументировать их универсальное назначение в реестре компонентов. Это проще и менее рискованно.

## 7. Обновление платформы (долгосрочно)

### 7.1 Как добавлять новые компоненты в платформу

```bash
# 1. В ветке main:
git checkout main
npm run create-section -- feature-cards
# Создаст: twig + css + js + обновит main.css и main.js

# 2. Разработать компонент, протестировать

# 3. Коммит в main

# 4. В каждом проекте:
git checkout project/retail-logistic
git merge main
# Новый компонент доступен — для активации добавить в JSON
```

### 7.2 Как обновлять ядро (PHP, Webpack, зависимости)

```bash
# 1. В ветке main:
git checkout main
composer update
npm update
# Проверить: npm run check

# 2. Коммит в main

# 3. В каждом проекте — merge main
```

### 7.3 Версионирование

| Событие | Версия | Пример |
|---------|--------|--------|
| Изменение ядра (PHP, маршрутизация, сборка) | major/minor | v2.0.0, v1.2.0 |
| Новый компонент/секция в библиотеке | minor | v1.3.0 |
| Исправления | patch | v1.2.1 |
| Изменения в проекте (данные, брендинг) | Тег проекта | kumho-v1.0.0, rl-v1.0.0 |

## 8. Будущее: PurgeCSS (опционально)

Если размер бандла станет критичным (сейчас — нет), можно добавить PurgeCSS в production-сборку:

```js
// postcss.config.js
module.exports = {
  plugins: [
    // ... существующие плагины
    process.env.NODE_ENV === 'production' && require('@fullhuman/postcss-purgecss')({
      content: ['templates/**/*.twig', 'data/json/**/*.json'],
      safelist: [/^swiper/, /^glightbox/, /^animate/]
    })
  ]
}
```

Это автоматически удалит неиспользуемые CSS-правила из бандла, не удаляя исходные файлы. Шаблоны остаются в проекте, но их стили не попадают в production CSS, если нет соответствующих HTML-элементов.

|--------|-------|
