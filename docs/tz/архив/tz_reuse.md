# Принципы и инфраструктура тиражируемости iSmart Platform

## Контекст и связь с основным ТЗ

Настоящий документ дополняет [основное ТЗ](tz_.md) — план трансформации iSmart Platform в тиражируемую платформу. Основное ТЗ описывает дорожную карту (Этапы 0-5) и детальный план Этапа 0 (отделение контента от ядра).

Данный документ формализует **принципы тиражируемости**, адаптированные под конкретную архитектуру iSmart Platform (PHP 8.5+, Slim 4, Twig 3, JSON-контент, DI-контейнер), и добавляет:
- Новые шаги 6-7 к Этапу 0
- Новый Этап 0.5 (инфраструктура тиражирования)
- Чеклисты проверки тиражируемости для каждого этапа

**Источник:** «Правила создания переиспользуемых и тиражируемых ИТ-решений» (20 правил, 6 групп).

## Принципы тиражируемости iSmart Platform

### I. Архитектура компонентов

**1. Единственная ответственность**
- Каждый Service отвечает за одну задачу: `DataLoaderService` — загрузка JSON, `SeoService` — SEO-данные, `LanguageService` — определение языка, `TemplateDataBuilder` — сборка данных для шаблона
- `PageAction` — оркестратор: вызывает сервисы и передаёт результат в Twig, но не содержит бизнес-логику
- При добавлении новой функциональности создаётся **новый сервис**, а не расширяется существующий

**2. Независимость**
- Middleware (PSR-15) — независимые компоненты: `CorrelationIdMiddleware`, `SecurityHeadersMiddleware`, `LanguageMiddleware` и др.
- Добавление/удаление middleware не требует правки других middleware или сервисов
- Взаимодействие — только через PSR-7 Request/Response (атрибуты, заголовки)

**3. Заменяемость**
- Все сервисы резолвятся через DI-контейнер (`config/container.php`)
- Замена реализации = замена определения в контейнере, не правка вызывающего кода
- Пример: замена `DataLoaderService` (JSON-файлы) на реализацию с БД — одно изменение в `container.php`

### II. Переиспользуемость

**4. Параметризация**
- Коллекции (`collections` в settings.php) — единый параметризованный механизм: `nav_slug`, `list_page_id`, `template`, `item_key`, `data_dir`, `slugs_source`, `og_type`, `extras_key`
- Новая коллекция = новая запись в конфиге, не новый метод в PHP
- Twig-компоненты (`templates/components/`) получают данные через параметры `{% include %}` / `{% embed %}`, не обращаются к глобальному состоянию

**5. Контекстная независимость**
- Twig-компоненты не предполагают конкретное окружение или заказчика
- Внешние зависимости (API-ключи, URL) подключаются через `.env` → `settings.php` → DI-контейнер
- JSON-контент не содержит путей к PHP-коду или ссылок на реализацию

**6. Версионирование**
- Семантическое версионирование (SemVer): `MAJOR.MINOR.PATCH`
- Процесс описан в `docs/guides/changelog-and-releases.md`
- `CHANGELOG.md` ведётся по формату Keep a Changelog
- Обратная совместимость обязательна при MINOR-обновлениях ядра

**7. Документированный контракт**
- Каждый public-метод сервисов в `src/Service/` имеет PHPDoc-контракт: `@param`, `@return`, `@throws`
- HTTP-контракты описаны в `docs/api/` (методы, обязательные поля, коды ответов)
- Twig-компоненты описаны в реестре компонентов (`docs/registry/components.md`)

### III. Тиражируемость

**8. Изоляция конфигурации — три уровня**

| Уровень | Файл | Что содержит | Кто редактирует |
|---------|------|-------------|----------------|
| Ядро | `config/settings.php` | Пути, Twig-конфигурация, CORS, rate limits, размеры изображений | Разработчик ядра |
| Проект | `config/project.php` | `route_map`, `collections`, `sitemap_pages`, интеграции | Команда заказчика |
| Окружение | `.env` | `APP_BASE_URL`, `APP_ENV`, API-ключи, метрика | DevOps / администратор |

**9. Multi-tenancy (стратегия)**

На текущем этапе — **single-tenant с отдельной инсталляцией на заказчика**:
- Каждый заказчик = своя копия платформы + свой контент + свой `.env`
- Ядро обновляется централизованно (git submodule или composer require)
- Данные изолированы физически (отдельные `data/json/` директории)

Переход к multi-tenancy (если потребуется) = добавление tenant identifier в middleware + фильтрация DataLoaderService по tenant. Архитектура позволяет это сделать без перестройки.

**10. Профили внедрения**

Профиль нового заказчика = набор файлов для конфигурации:

```
config/project.php          — конфигурация (на основе project.php.dist)
data/json/{lang}/pages/     — JSON-контент страниц
data/json/global.json       — навигация, контакты, языки
assets/css/base/variables.css — цветовая схема, шрифты, логотип
.env                        — окружение
```

Создание нового внедрения: скрипт `tools/scaffold/create-project.sh` генерирует эту структуру с дефолтными значениями.

**11. Минимальное вмешательство**

- Адаптация под нового заказчика — **без изменения PHP-кода ядра**
- Расширение через: конфигурацию (`project.php`), контент (`data/json/`), стили (`variables.css`), переопределение шаблонов (`templates/override/`)
- Если нужна кастомная логика — через Event Dispatcher (PSR-14), не модификацию ядра

### IV. Глобальные решения

**12. Платформенное мышление**

Чёткое разделение:

| Слой | Содержимое | Обновляется |
|------|-----------|-------------|
| **Ядро платформы** | `src/`, `templates/sections/`, `templates/components/`, `config/settings.php`, `tools/` | Централизованно (одна команда) |
| **Проектный слой** | `data/json/`, `config/project.php`, `assets/css/base/variables.css`, `templates/override/` | Для каждого заказчика отдельно |

**13. Реестр компонентов**

Ведётся каталог `docs/registry/components.md` — все переиспользуемые Twig-компоненты:
- Имя компонента, путь к файлу
- Принимаемые параметры (с типами и значениями по умолчанию)
- Пример использования
- Статус (stable / experimental)

Обязателен поиск в реестре перед созданием нового компонента.

**14. Управляемый техдолг**

- Оставшиеся хардкоды фиксируются в `docs/platform-content-separation.md` (раздел «Оставшиеся точки хардкода»)
- Каждый элемент: описание, приоритет, плановый этап устранения
- Техдолг не блокирует тиражирование, но планируется к устранению

### V. Жизненный цикл

**15. Сопровождаемость**

Каждая инсталляция поставляется с:
- `docs/guides/deploy-checklist.md` — чеклист продакшн-выкатки (25 пунктов)
- `docs/guides/deploy-ftp.md` — инструкция по развёртыванию на хостинге
- `CHANGELOG.md` — журнал изменений
- `.env.example` — шаблон окружения с описанием переменных

**16. Наблюдаемость**

| Компонент | Реализация | Статус |
|-----------|-----------|--------|
| Логи | Monolog → `logs/app.log` | Реализовано |
| Request ID | `CorrelationIdMiddleware` → заголовок `X-Request-Id` | Реализовано |
| JSON-формат логов | timestamp, level, request_id, duration_ms, method, path | Этап 0.5 |
| Duration | Middleware для замера длительности запроса | Этап 0.5 |
| Аналитика | Яндекс.Метрика, Roistat, Top.Mail.Ru | Реализовано |

Стандарт наблюдаемости — единый для всех инсталляций.

**17. Тестируемость**

| Тип | Инструмент | Покрытие |
|-----|-----------|---------|
| Unit | PHPUnit 10 | DataLoaderService, LanguageService, SeoService, TemplateDataBuilder, JsonProcessor |
| Integration | PHPUnit 10 | PageAction, Middleware |
| JS | Vitest | debounce-throttle, url, validation |
| Smoke | Node.js | Основные маршруты и редиректы |
| JSON validation | npm script | Валидация структуры JSON-контента |

Порог покрытия ядра: **>= 80%**. Тесты хранятся вместе с кодом (`tests/`).

### VI. Организационные правила

**18. Владение**
- Ядро платформы = одна команда/владелец, отвечает за развитие, совместимость, SemVer
- Проектный контент = команда заказчика, отвечает за наполнение, брендинг

**19. Внутренний рынок**
- Инсталляции потребляют ядро через контракты (API сервисов, Twig-интерфейсы, конфигурация)
- Форк ядра — только с обоснованием; предпочтительнее расширение через Event Dispatcher

**20. Приоритет переиспользования**

Порядок принятия решения при новой задаче:
1. Найти готовый компонент в реестре (`docs/registry/components.md`)
2. Адаптировать существующий (параметры, конфигурация)
3. Запросить доработку у владельца ядра
4. Создать новый компонент (с регистрацией в реестре)

## Архитектура тиражирования

### Целевая структура инсталляции

```
Инсталляция заказчика:
├── core/                              # Ядро: git submodule или composer require
│   ├── src/                           #   PHP-сервисы, Actions, Middleware, Twig Extensions
│   ├── templates/                     #   Базовые шаблоны (base.twig, sections/, components/)
│   ├── config/settings.php            #   Настройки ядра
│   └── tools/                         #   Scaffold-скрипты, утилиты
│
├── config/
│   └── project.php                    # Конфигурация заказчика (из project.php.dist)
│
├── data/json/
│   ├── global.json                    # Навигация, контакты, языки
│   └── {lang}/pages/                  # Контент страниц
│
├── assets/css/base/
│   └── variables.css                  # Брендинг: цвета, шрифты, логотип
│
├── templates/override/                # Переопределения шаблонов (если нужно)
│
├── .env                               # Окружение
└── public/index.php                   # Точка входа
```

### Ключевое решение

Twig-шаблоны ядра спроектированы как **data-driven** — получают все данные из JSON через `TemplateDataBuilder`. В большинстве случаев переопределение шаблонов **не требуется**: достаточно подставить другой JSON-контент и CSS-переменные.

### Расширенная таблица этапов

| Этап | Что | Результат | Принцип тиражируемости |
|------|-----|-----------|----------------------|

| **0. Шаг 6** | `config/project.php` — вынос проектных настроек | Три уровня конфигурации | Изоляция конфигурации (#8) |

| **0.5 Инфраструктура тиражирования** | Контракты, Event Dispatcher, реестр, scaffold, логи | Готовность к тиражированию | Контракт (#7), Заменяемость (#3), Реестр (#13), Наблюдаемость (#16) |
| **1. Контент «РЛ»** | Новые страницы, ребрендинг | Первое тиражирование | Профили внедрения (#10), Минимальное вмешательство (#11) |
| **2. Аутентификация** | Регистрация/вход, модерация | Авторизованные пользователи | Расширяемость через Event Dispatcher, не хардкод |

| **5. АРМ и инструменты** | Справочники, AI-агенты, сервисы | Рабочие инструменты | АРМ-модули = компоненты реестра |

### Шаг 6 Этапа 0: config/project.php

**Файл:** `config/settings.php` (строки ~74-103)

Выделить из `settings.php` проектно-специфичные данные в отдельный файл:

**config/project.php** (проект заказчика):
```php
<?php
return [
    'route_map' => [
        'tires' => 'tires-list',
    ],
    'collections' => [
        'tires' => [
            'nav_slug'     => 'tires',
            'list_page_id' => 'tires-list',
            'template'     => 'pages/tire.twig',
            'item_key'     => 'item',
            'data_dir'     => 'tires',
            'slugs_source' => 'items',
            'og_type'      => 'website',
            'extras_key'   => 'tire',
        ],
        // ... остальные коллекции
    ],
    'sitemap_pages' => ['index', 'contacts', 'policy', 'agree', 'tires-list', 'news'],
    'integrations' => [
        'photoroom' => [
            'enabled' => true,
        ],
    ],
];
```

**config/project.php.dist** (шаблон для нового заказчика):
```php
<?php
// Конфигурация проекта заказчика
// Скопируйте в project.php и адаптируйте
return [
    'route_map' => [],
    'collections' => [],
    'sitemap_pages' => ['index', 'contacts', 'policy', 'agree'],
    'integrations' => [],
];
```

**Изменение в settings.php:**
```php
// Загрузка проектной конфигурации
$projectConfig = [];
$projectConfigPath = __DIR__ . '/project.php';
if (file_exists($projectConfigPath)) {
    $projectConfig = require $projectConfigPath;
}

// Мердж с настройками ядра
return [
    // ... настройки ядра ...
    'route_map'     => $projectConfig['route_map'] ?? [],
    'collections'   => $projectConfig['collections'] ?? [],
    'sitemap_pages' => $projectConfig['sitemap_pages'] ?? ['index'],
    'integrations'  => $projectConfig['integrations'] ?? [],
];
```

## Этап 0.5: Инфраструктура тиражирования

Промежуточный этап между отделением контента (Этап 0) и первым внедрением (Этап 1).

### 0.5.1 Контракты сервисов

**Цель:** Документировать API Service Layer для возможности подмены реализаций через DI.

**Действия:**
- Добавить PHPDoc-контракты на все public-методы `DataLoaderService`, `SeoService`, `TemplateDataBuilder`, `LanguageService`
- Создать PHP-интерфейсы для ключевых сервисов (опционально, если потребуется подмена)
- Обновить `docs/api/` — добавить раздел Service Layer

**Файлы:** `src/Service/*.php`, `docs/api/services.md` (новый)

### 0.5.2 Event Dispatcher (PSR-14)

**Цель:** Расширяемость без модификации ядра.

**Действия:**
- Подключить PSR-14 Event Dispatcher (например, `league/event` через composer)
- Создать события: `PageLoaded`, `EntityResolved`, `SeoBuilt`
- Интегрировать в `PageAction`: диспатчить события в ключевых точках
- Регистрация слушателей — в `config/project.php` или отдельном `config/listeners.php`

**Файлы:**
- `src/Event/PageLoaded.php`, `EntityResolved.php`, `SeoBuilt.php` (новые)
- `src/Action/PageAction.php` (диспатч событий)
- `config/container.php` (регистрация EventDispatcher)

**Пример использования (заказчик):**
```php
// config/listeners.php
return [
    PageLoaded::class => [
        function (PageLoaded $event) {
            // Кастомная логика заказчика
        },
    ],
];
```

### 0.5.3 Реестр компонентов

**Цель:** Каталог переиспользуемых Twig-компонентов.

**Действия:**
- Создать `docs/registry/components.md`
- Описать каждый компонент из `templates/components/`: имя, путь, параметры, пример, статус
- Описать каждую секцию из `templates/sections/`: header, footer, intro, content, contacts, cookie-panel

**Формат записи:**
```markdown
### form-callback
- **Путь:** `templates/components/form-callback.twig`
- **Параметры:** `title` (string), `fields` (array), `action` (string), `method` (string, default: "POST")
- **Пример:** `{% include 'components/form-callback.twig' with { title: 'Обратный звонок', ... } %}`
- **Статус:** stable
```

### 0.5.4 Scaffold-скрипт

**Цель:** Автоматизация создания нового проекта.

**Действия:**
- Создать `tools/scaffold/create-project.sh`
- Скрипт принимает имя проекта, создаёт структуру каталогов
- Копирует `project.php.dist` → `project.php`
- Создаёт минимальный `data/json/` с дефолтным контентом (index, contacts, policy, agree)
- Создаёт `assets/css/base/variables.css` с плейсхолдерами брендинга
- Создаёт `.env` из `.env.example`

**Файл:** `tools/scaffold/create-project.sh` (новый)

### 0.5.5 Расширение наблюдаемости

**Цель:** Логи пригодны для агрегации (ELK, Loki).

**Действия:**
- Переключить Monolog на JSON-формат (JsonFormatter или кастомный)
- Добавить middleware для замера длительности запроса (`RequestDurationMiddleware`)
- Обогатить контекст логов: `timestamp`, `level`, `request_id`, `duration_ms`, `method`, `path`

**Файлы:**
- `config/container.php` (настройка JsonFormatter)
- `src/Middleware/RequestDurationMiddleware.php` (новый)
