Настоящий документ расширяет фреймворк измерений из [tz_dimensions.md](tz_dimensions.md), добавляя **четвёртую группу** — **Идентичность**.

- [tz_.md](tz_.md) — план этапов трансформации платформы (Этапы 0-5)

| 8 | **Персональность** | Система знает, кто ты? | ОТНОШЕНИЕ К ПОЛЬЗОВАТЕЛЮ | АРМ адаптируется под конкретного сотрудника |

### Ключевое разграничение

```
Индивидуальность — свойство САМОЙ системы
Персональность   — отношение системы К ПОЛЬЗОВАТЕЛЮ

Snowflake — индивидуальна (каждая снежинка уникальна)
Но она не персональна — ей всё равно, кто смотрит

Официант в ресторане — персонален (помнит твои предпочтения)
Но не индивидуален — таких официантов много
```

**Индивидуальность (Individuality)** — способность системы иметь собственный характер, уникальный набор свойств, отличающий её от других экземпляров той же платформы. Достигается через:

1. **Уникальный контекст** — система создана под конкретные условия и не претендует на другие
2. **Внутренняя целостность** — все части согласованы вокруг одной идеи (бизнес-домен)
3. **Намеренная специализация** — осознанный выбор ниши и отказ от обобщения
4. **Неповторимое происхождение** — история создания, данные, workflow'ы встроены в продукт

**Персональность (Personalization)** — способность системы адаптироваться под конкретного пользователя, помнить его контекст и менять поведение на основе его предпочтений. Достигается через:

1. **Память о пользователе** — система накапливает историю взаимодействий
2. **Адаптация поведения** — интерфейс, рекомендации, порядок элементов меняются под человека
3. **Контекстуальная чувствительность** — система реагирует на ситуацию пользователя, не только на команды
4. **Эмпатическая модель** — система строит модель целей и состояний конкретного человека

```
│  Тиражируемость ─────── новый клиент = легко              │
│  Автономность ────────── работает сама                     │
│  Масштабирование ────── не упадёт                         │
└──────────────────────────────────────────────────────────┘

┌──────────────────────────────────────────────────────────┐
│                      ГИБКОСТЬ                            │
│  «Делай разное и по-разному»                             │
│                                                          │
│  Расширяемость, Конфигурируемость                         │

└──────────────────────────────────────────────────────────┘

┌──────────────────────────────────────────────────────────┐
│                   УНИВЕРСАЛЬНОСТЬ                        │

│  Абстракция, Стандартизация, Обобщение                    │
└──────────────────────────────────────────────────────────┘

┌──────────────────────────────────────────────────────────┐
│                    ИДЕНТИЧНОСТЬ                           │
│  «Делай МОЁ и МЕНЯ»                                     │
│                                                          │
│  Индивидуальность ──── система знает, кто она             │
│  Персональность ────── система знает, кто ты              │
└──────────────────────────────────────────────────────────┘
```

**Четыре вектора тянут систему в разные стороны:**

```
              Индивидуальность
                     ▲
                     │
                     │
Персональность ◄─────┼─────► Универсальность
                     │
                     │
                     ▼
                 Гибкость
                     │
                     ▼
               Эффективность
```

Эффективность требует стандартизации. Гибкость требует вариативности. Универсальность требует обобщения. Индивидуальность требует уникальности. Персональность требует специализации под одного. **Пять разных векторов — и осознанный выбор баланса между ними.**

## Парадоксы Индивидуальности

### Индивидуальность vs Переиспользуемость

```
Переиспользуемость говорит:     Индивидуальность говорит:
«Сделай общее — используй       «Я создан для этого момента,
везде»                           этого места, этой цели»
```

**Конфликт:** Абстракция (`BaseAgent`, `loadEntity`) делает код универсальным, но стирает уникальность. Handmade-нож кузнеца индивидуален; массовое производство = переиспользование логики, но потеря индивидуальности.

**Решение в iSmart Platform:**

Разделение на **фиксированный контракт** и **индивидуальную реализацию**:

```
BaseAgent (переиспользуемый)          LabelExtractorAgent (индивидуальный)
├── validate() — фиксированный         ├── process() — полностью уникальный
├── run() — фиксированный pipeline     ├── _extract_from_image() — своя логика
└── AgentInput/Output — контракт       └── Промпт — под домен алкоголя
```

- `BaseAgent`, `loadEntity`, DI-контейнер — переиспользуемы
- `process()`, MCP Prompts, стратегии MAS — индивидуальны для каждого deployment
- Переиспользуемость = на уровне контракта. Индивидуальность = на уровне реализации

### Индивидуальность vs Тиражируемость

```
Тиражируемость говорит:          Индивидуальность говорит:
«Копируй без потерь качества»    «Копия — уже не я»
```

**Конфликт:** Scaffold-скрипт создаёт новую инсталляцию за минуту. Но если все инсталляции идентичны — где индивидуальность? Картина Ван Гога индивидуальна; печатная копия тиражируема, но индивидуальность оригинала не копируется.

**Решение в iSmart Platform:**

Тиражируется **ядро**, индивидуальность — в **проектном слое**:

```
solution/
├── core/                           # ТИРАЖИРУЕМОЕ — одинаковое у всех
│   ├── php-site/src/               #   PHP-сервисы, Actions, Middleware
│   ├── django-agents/              #   BaseAgent, API
│   └── n8n-workflows/              #   Базовые workflow'ы
│
├── deployments/
│   ├── retail-logistic/            # ИНДИВИДУАЛЬНОЕ — уникальное для РЛ
│   │   ├── config/project.php      #   Роли: MDM, закупка/ВЭД
│   │   ├── data/json/              #   Справочники алкогольной продукции
│   │   ├── assets/css/variables.css#   Фирменный стиль РЛ
│   │   └── docker-compose.yml      #   Инфраструктура РЛ
│   │
│   └── tire-company/               # ИНДИВИДУАЛЬНОЕ — уникальное для шинного бренда
│       ├── config/project.php      #   Роли: менеджер, склад
│       ├── data/json/              #   Каталог шин
│       └── assets/css/variables.css#   Фирменный стиль бренда
│
└── extensions/
    └── retail-logistic/            # ИНДИВИДУАЛЬНОЕ — кастомные расширения
        ├── custom-agents/          #   wine_classifier_agent.py
        ├── custom-workflows/       #   ved-document-processing.json
        └── custom-templates/       #   arm-purchasing.twig
```

**Принцип:** Ядро тиражируется. Характер не тиражируется — он создаётся для каждого deployment.

### Индивидуальность vs Масштабирование

```
Масштабирование говорит:         Индивидуальность говорит:
«Больше единиц — те же           «Каждая единица — особая»
свойства»
```

**Конфликт:** При масштабировании индивидуальность размывается до статистики. 1 000 000 «уникальных» элементов — это уже не индивидуальность, а вариативность.

**Решение в iSmart Platform:**

Индивидуальность на уровне **deployment**, не на уровне **экземпляра**:

| Уровень | Масштабируется? | Индивидуален? | Пояснение |
|---------|:-:|:-:|-----------|
| Контейнер (Docker replica) | Да | Нет | Реплики идентичны — это масштабирование |
| Deployment (инсталляция клиента) | Нет | Да | Каждый deployment уникален |
| Тенант (multi-tenant) | Да | Частично | Общий код, но уникальные данные и конфиг |

Масштабирование = горизонтальное копирование внутри deployment. Индивидуальность = вертикальная уникальность между deployment'ами.

## Парадоксы Персональности

### Персональность vs Универсальность

```
Универсальность говорит:         Персональность говорит:
«Работает для всех одинаково»    «Работает для тебя иначе,
                                  чем для него»
```

**Конфликт:** MCP Server предоставляет стандартный интерфейс для любой AI-модели. Но ответы должны зависеть от того, кто спрашивает.

**Решение в iSmart Platform:**

Универсальный **протокол** + персональный **контекст**:

```python
# MCP Server — универсальный интерфейс
@server.resource("tasks://history")
async def task_history(user_id: str, limit: int = 10) -> str:
    """История задач — одинаковый интерфейс, персональные данные."""
    # Протокол универсален: любая AI-модель вызывает одинаково
    # Данные персональны: возвращается история КОНКРЕТНОГО пользователя
    tasks = await db.get_tasks(user_id=user_id, limit=limit)
    return json.dumps(tasks)

# MCP Prompts — role-aware
@server.prompt("label-analysis")
async def label_analysis_prompt(user_role: str) -> str:
    """Системный промпт учитывает роль пользователя."""
    base = "Ты — эксперт по алкогольной продукции."
    if user_role == "mdm":
        return f"{base} Фокусируйся на точности мастер-данных и классификации."
    elif user_role == "purchasing_ved":
        return f"{base} Фокусируйся на соответствии документам ВЭД и ценообразовании."
    return base
```

**Принцип:** Протокол (MCP, REST API) — универсальный. Контекст (user_id, role, history) — персональный. Они сосуществуют **на разных слоях**.

### Персональность vs Автономность

```
Автономность говорит:            Персональность говорит:
«Я сам принимаю решения          «Мои решения зависят от тебя»
по своим правилам»
```

**Конфликт:** Автономный AI-агент решает, как классифицировать продукт. Но осторожный MDM-специалист хочет подтверждать каждое решение, а опытный — доверяет автоматике.

**Решение в iSmart Platform:**

Автономность **в рамках** персонального контекста:

```php
// config/project.php — роли определяют базовый уровень автономности
'roles' => [
    'mdm' => [
        'label' => 'MDM',
        'autonomy_level' => 2,           // полуавтономный: система предлагает, человек подтверждает
        'confidence_threshold' => 0.9,    // высокий порог: автоматически только при 90%+
    ],
    'purchasing_ved' => [
        'label' => 'Закупка и ВЭД',
        'autonomy_level' => 1,           // инструментальный: человек использует инструменты
        'confidence_threshold' => 0.95,   // очень высокий: почти всё требует подтверждения
    ],
    'admin' => [
        'label' => 'Администратор',
        'autonomy_level' => 3,           // автономный: система решает сама, человек контролирует
        'confidence_threshold' => 0.7,    // низкий: доверяет автоматике
    ],
],
```

```python
# agents-service — агент учитывает персональный контекст
class LabelExtractorAgent(BaseAgent):
    def process(self, data: AgentInput) -> AgentOutput:
        result = self._extract_from_image(data.payload)

        # Автономность зависит от персонального порога
        threshold = data.payload.get('confidence_threshold', 0.8)
        if result['confidence'] >= threshold:
            result['auto_approved'] = True   # автоматическое решение
        else:
            result['auto_approved'] = False  # требуется подтверждение человека
            result['reason'] = f"Confidence {result['confidence']:.0%} < threshold {threshold:.0%}"

        return AgentOutput(
            status="success",
            result=result,
            confidence=result['confidence'],
        )
```

**Принцип:** Автономность не фиксирована — она **параметризована** пользовательским контекстом. Агент решает «сам», но в рамках, заданных конкретным человеком.

### Персональность vs Масштабирование

```
Масштабирование говорит:         Персональность говорит:
«Обслужи миллион пользователей»  «Помни каждого из миллиона»
```

**Конфликт:** Это самый дорогой компромисс в IT. Netflix персонализирует рекомендации для 300 млн пользователей — стоимость инфраструктуры персонализации = сотни миллионов долларов в год.

**Решение в iSmart Platform:**

**Слоистая персонализация** — каждый слой добавляет стоимость, но подключается **опционально**:

```
┌─────────────────────────────────────────────────┐
│ Уровень 4: AI-модель пользователя               │  Дорого, опционально
│ «AI учится на твоих решениях»                    │  Этап 5+ (будущее)
│ Стоимость: высокая (модель на пользователя)      │
├─────────────────────────────────────────────────┤
│ Уровень 3: История взаимодействий               │  Средне, lazy-load
│ «Система помнит твои прошлые задачи»             │  Этап 4-5
│ Стоимость: средняя (запись в Django Core)         │
├─────────────────────────────────────────────────┤
│ Уровень 2: Настройки пользователя               │  Дёшево
│ «Предпочтения, избранное, порядок виджетов»      │  Этап 3
│ Стоимость: низкая (JSON в user profile)          │
├─────────────────────────────────────────────────┤
│ Уровень 1: Роль                                 │  Минимальная
│ «MDM видит одно, закупщик — другое»              │  Этап 2-3
│ Стоимость: минимальная (3-5 конфигов ролей)      │
└─────────────────────────────────────────────────┘
```

| Уровень | Что хранится | Где хранится | Стоимость на пользователя | Этап |
|---------|-------------|-------------|:-------------------------:|:----:|
| 1. Роль | Набор доступных АРМ | `project.php` | ~0 (общий конфиг) | 2-3 |
| 2. Настройки | Избранное, порядок, тема | Django Core (UserProfile) | ~1 KB | 3 |
| 3. История | Последние задачи, решения | Django Core (TaskHistory) | ~100 KB/мес | 4-5 |
| 4. AI-модель | Персональные веса, предпочтения | Django Agents | ~10 MB | 5+ |

**Принцип:** Персонализация масштабируется через слои. Уровень 1-2 = почти бесплатно для 1000 пользователей. Уровень 3-4 = дорого, включается только когда оправдано.

## Индивидуальность в iSmart Platform — конкретные механизмы

### Что делает каждый deployment уникальным

| Уровень | Что делает систему индивидуальной | Механизм | Файлы/конфигурация |
|---------|----------------------------------|----------|-------------------|
| Бизнес-домен | Отрасль, терминология, процессы | Данные + агенты | `data/json/`, `extensions/custom-agents/` |
| Брендинг | Визуальная идентичность | CSS-переменные + global.json | `variables.css`, `data/json/global.json` |
| Процессы | Уникальные бизнес-процессы | n8n workflow'ы | `extensions/custom-workflows/` |
| AI-характер | Доменные промпты и стратегии | MCP Prompts + MAS стратегии | `mcp_server.py`, `strategies/` |
| Ролевая модель | Организационная структура | project.php | `config/project.php` → `roles` |
| Данные | Справочники, классификаторы | Django Core models + JSON | `data/json/`, Django migrations |

### Пример: индивидуальность deployment'а «Ритейл Логистик»

```
Deployment «Ритейл Логистик» — ИНДИВИДУАЛЕН:

Бизнес-домен:
  - Алкогольная продукция (вино, виски, водка, ликёры)
  - Терминология: акциз, ЕГАИС, этикетка, контрэтикетка
  - Справочники: производители, регионы происхождения, сорта винограда

Агенты (индивидуальные):
  - LabelExtractorAgent: обучен на этикетках алкоголя
  - WineClassifierAgent: классификация по типу, региону, сорту
  - VedDocumentAgent: работа с документами ВЭД для алкоголя

MCP Prompts (индивидуальные):
  - «Ты — эксперт по алкогольной продукции»
  - «Классифицируй по ОКПД 2 для алкоголя»

Workflow'ы (индивидуальные):
  - ved-document-processing: обработка ВЭД-документов
  - excise-verification: проверка акцизных марок

Роли (индивидуальные):
  - MDM: управление мастер-данными алкогольной продукции
  - Закупка/ВЭД: работа с поставщиками и таможней
```

### Как индивидуальность сочетается с тиражируемостью

```
                    ЯДРО (тиражируемое)
                         │
          ┌──────────────┼──────────────┐
          │              │              │
     РЛ (алкоголь)  Шины (авто)   Будущий клиент
          │              │              │
    ┌─────┤         ┌────┤         ┌────┤
    │     │         │    │         │    │
  Агенты  │       Агенты │       Агенты │
  Промпты │       Промпты│       Промпты│
  Данные  │       Данные │       Данные │
  Стиль   │       Стиль  │       Стиль  │
  Роли    │       Роли   │       Роли   │
    │     │         │    │         │    │
    └─ ИНДИВИДУАЛЬНОСТЬ ─┘        └────┘
```

**Метафора:** Ядро — это **ДНК**. Deployment — это **организм**. ДНК одинакова (тиражируема), но каждый организм уникален благодаря окружению, данным и опыту.

## Персональность в iSmart Platform — конкретные механизмы

### Что адаптируется под конкретного пользователя

| Уровень | Что адаптируется | Механизм | Где хранится |
|---------|-----------------|----------|-------------|
| Роль | Набор доступных АРМ-модулей | `project.php` → `roles[].arms[]` | `config/project.php` |
| Dashboard | Виджеты, порядок, компоновка | User preferences API | Django Core (UserProfile) |
| Инструменты | Избранные, недавние, закреплённые | Pinned tools, recent history | Django Core (UserPreferences) |
| AI-контекст | Промпты учитывают роль и опыт | MCP Prompts с user context | MCP Server + JWT claims |
| Уведомления | Каналы (email, Telegram, push), частота | Notification settings per user | Django Core (NotificationSettings) |
| Пороги | confidence_threshold, уровень автономности | Per-user overrides | Django Core (UserProfile) |
| Интерфейс | Тема (светлая/тёмная), язык, формат дат | User settings | PHP session / cookie |

## Классификация систем

### 4 типа по осям идентичности

| Тип | Знает себя? | Знает тебя? | Характеристика |
|-----|:-----------:|:-----------:|---------------|
| **Инструмент** | Нет | Нет | Просто работает. grep, curl, калькулятор |
| **Характерная система** | Да | Нет | Имеет собственный стиль, философию. vim, R, NeXT |
| **Сервисная система** | Нет | Да | Подстраивается под тебя, но не имеет «характера». Netflix, Spotify |
| **Живая система** | Да | Да | Знает себя И тебя. iPhone + Siri (с оговорками) |

### Путь iSmart Platform

```
Текущее        Этап 1           Этап 2-3         Этап 4-5
состояние      (deployment РЛ)  (роли, профиль)  (AI + MCP)

ИНСТРУМЕНТ  →  ХАРАКТЕРНАЯ   →  (промежуточное) → ЖИВАЯ
                СИСТЕМА          состояние         СИСТЕМА

Не знает       Знает себя:      Начинает         Знает себя:
ни себя,       «Я — платформа   узнавать тебя:   домен, агенты,
ни тебя        РЛ для алкоголя, «Ты — MDM или    промпты.
               мой стиль,       закупщик,         Знает тебя:
               мои данные»      твои настройки»   история,
                                                   предпочтения,
                                                   пороги
```

### Почему «Живая система» — целевое состояние

**Живая система** не означает AGI или самосознание. Это означает:
1. Система **отличается** от других deployment'ов (индивидуальность) — не просто другой конфиг, а другой характер, другие агенты, другие промпты
2. Система **помнит** каждого пользователя (персональность) — не просто роль, а историю, предпочтения, контекст
3. Система **растёт** с каждым взаимодействием — история обогащает AI-контекст
4. Система **адаптируется** — пороги доверия, уровень автономности меняются на основе опыта

## Расширенная матрица компромиссов

Дополнение к матрице из [tz_dimensions.md](tz_dimensions.md):

| | Переисп. | Тираж. | Автоном. | Масштаб. | Универс. | Гибкость | **Индив.** | **Персон.** |
|---|:---:|:---:|:---:|:---:|:---:|:---:|:---:|:---:|
| **Переисп.** | — | + | ~ | ~ | + | - | **-** | ~ |
| **Тираж.** | + | — | ~ | ~ | - | ~ | **-** | ~ |
| **Автоном.** | ~ | ~ | — | + | + | - | ~ | **-** |
| **Масштаб.** | ~ | ~ | + | — | ~ | - | **-** | **-** |
| **Универс.** | + | - | + | ~ | — | ~ | ~ | **-** |
| **Гибкость** | - | ~ | - | - | ~ | — | + | + |
| **Индив.** | **-** | **-** | ~ | **-** | ~ | + | — | ~ |
| **Персон.** | ~ | ~ | **-** | **-** | **-** | + | ~ | — |

`+` усиливают, `-` конфликтуют, `~` нейтральны

**Наблюдения:**
- **Индивидуальность** конфликтует с Переиспользуемостью, Тиражируемостью и Масштабированием — с тремя из четырёх измерений Эффективности. Это закономерно: эффективность требует стандартизации, индивидуальность — уникальности
- **Персональность** конфликтует с Универсальностью, Автономностью и Масштабированием — с одним измерением из каждой группы
- **Гибкость** усиливает оба новых измерения — гибкая система может быть и индивидуальной, и персональной
- **Индивидуальность и Персональность** нейтральны друг к другу — можно иметь одно без другого
