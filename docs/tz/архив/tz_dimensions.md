Настоящий документ — **мета-фреймворк**, связывающий все ТЗ проекта через 6 измерений системы.

**Связанные документы:**

**Проблема:** Существующие ТЗ детально описывают *что делать*, но не формализуют *по каким осям* оценивать систему. Без чёткого разделения измерений возникают ошибочные допущения:
- «Сделаем переиспользуемым = будет масштабироваться» (нет!)
- «Тиражируемость = универсальность» (нет!)
- «Автономные агенты = гибкая система» (не обязательно!)

**Цель:** Формализовать 6 измерений, показать покрытие каждого измерения в существующих ТЗ, выявить пробелы и дать рекомендации.

## 6 измерений системы

Каждое измерение отвечает на один вопрос:

| # | Измерение | Вопрос | Фокус | Пример в системе |
|---|-----------|--------|-------|-----------------|
| 1 | **Переиспользуемость** | Можно взять повторно? | КОД | `BaseAgent` используется всеми агентами |
| 2 | **Тиражируемость** | Можно развернуть заново? | ПРОДУКТ | `create-project.sh` → новая инсталляция |
| 3 | **Автономность** | Работает самостоятельно? | ПОВЕДЕНИЕ | AI-агент сам выбирает инструменты через MCP |
| 4 | **Масштабирование** | Выдержит рост? | НАГРУЗКА | 3 Django-сервиса на разных портах |
| 5 | **Универсальность** | Работает в любом контексте? | АБСТРАКЦИЯ | MCP-сервер — любая AI-модель подключается |
| 6 | **Гибкость** | Легко меняется? | АДАПТАЦИЯ | DI-контейнер, Event Dispatcher, `project.php` |

### Определения

**Переиспользуемость (Reusability)** — способность кода быть использованным в разных местах без копирования. Снижает дублирование, повышает качество. Реализуется через абстракции, параметризацию, компонентную архитектуру.

**Тиражируемость (Replicability)** — способность продукта быть развёрнутым для нового заказчика с минимальными усилиями. Не про код, а про весь продукт: конфигурация, данные, брендинг, инфраструктура.

**Автономность (Autonomy)** — способность системы или её части действовать самостоятельно, принимая решения без вмешательства человека. Спектр от ручного управления до полной автоматизации.

**Масштабирование (Scalability)** — способность системы обрабатывать растущую нагрузку. Горизонтальное (больше экземпляров) и вертикальное (мощнее сервер).

**Универсальность (Universality)** — способность системы работать в любом контексте, с любыми клиентами, инструментами, данными. Достигается через стандартизацию интерфейсов и обобщение.

**Гибкость (Flexibility)** — способность системы адаптироваться к изменениям без существенной перестройки. Расширяемость, конфигурируемость, слабая связанность.

## Три группы измерений

```
┌──────────────────────────────────────────────────────────┐
│                     ЭФФЕКТИВНОСТЬ                        │
│  «Хорошо делает то, для чего создана»                    │
│                                                          │
│  Переиспользуемость ─── не пиши дважды                   │
│  Тиражируемость ─────── новый клиент = легко              │
│  Автономность ────────── работает сама                     │
│  Масштабирование ────── не упадёт                         │
└──────────────────────────────────────────────────────────┘

┌──────────────────────────────────────────────────────────┐
│                      ГИБКОСТЬ                            │
│  «Может делать то, для чего ещё не создавалась»          │
│                                                          │
│  Расширяемость ──────── Event Dispatcher, плагины         │
│  Конфигурируемость ──── project.php, .env                 │
│  Слабая связанность ─── DI-контейнер, интерфейсы          │
│  Инверсия зависимостей ─ контракты, не реализации          │
└──────────────────────────────────────────────────────────┘

┌──────────────────────────────────────────────────────────┐
│                   УНИВЕРСАЛЬНОСТЬ                        │
│  «Работает в любом контексте»                            │
│                                                          │
│  Абстракция ─────────── единый контракт (BaseAgent)       │
│  Стандартизация ──────── OpenAPI, MCP, JSON-логи           │
│  Обобщение ──────────── loadEntity вместо loadTire         │
└──────────────────────────────────────────────────────────┘
```

**Связи между группами:**
- Эффективность фокусируется на текущих задачах → «делать правильные вещи правильно»
- Гибкость фокусируется на будущих задачах → «быть готовым к неизвестному»
- Универсальность фокусируется на контексте → «работать где угодно с кем угодно»

### 3. Автономность — «Работает самостоятельно?»

**Ключевой пробел.** Автономность — наименее проработанное измерение в текущих ТЗ.

**Что уже есть / запланировано:**

| Решение | Где описано | Статус |
|---------|-----------|--------|
| `BaseAgent` с pipeline validate → process → return | [tz_architecture.md](tz_architecture.md) | Планируется (Этап 4-5) |
| MAS Orchestrator — координация нескольких агентов | [tz_architecture.md](tz_architecture.md) | Планируется (Этап 5) |
| MCP — AI-модель сама обнаруживает инструменты | [tz_protocols.md](tz_protocols.md) | Планируется (Этап 4-5) |
| n8n workflow'ы — автоматизация процессов по расписанию | [tz_architecture.md](tz_architecture.md) | Планируется (Этап 4) |

**Чего не хватает:**

**Спектр автономности** — нет чёткой модели перехода от ручного к полностью автоматическому:

```
Уровень 0        Уровень 1          Уровень 2          Уровень 3          Уровень 4
РУЧНОЙ            ИНСТРУМЕНТ          ПОЛУАВТОНОМНЫЙ      АВТОНОМНЫЙ          САМОУПРАВЛЯЕМЫЙ

Человек           Человек             Система             Система             Система
делает всё        использует          предлагает,         решает сама,        улучшает
                  инструменты         человек             человек             себя сама
                                      подтверждает        контролирует

Пример:           Пример:             Пример:             Пример:             Пример:
Ручной ввод       AI извлекает        AI предлагает       AI сама             AI оптимизирует
данных с          данные,             классификацию,      классифицирует      pipeline на
этикетки          человек             человек             и сохраняет,        основе
                  проверяет           утверждает          человек             статистики
                                                          аудитит             ошибок
```

**Для каждого АРМ и инструмента нужно определить:**
- Текущий уровень автономности
- Целевой уровень
- Условия перехода (критерий accuracy, объём данных для обучения)

**Не хватает:**
- Стратегия перехода по уровням для каждого инструмента
- Human-in-the-loop паттерн: как реализовать «утверждение» и «аудит» в PHP-интерфейсе
- Метрики доверия: при какой accuracy (confidence) агент переходит на следующий уровень
- Fallback-стратегия: что делать, когда агент не уверен (confidence < порога)

**Рекомендация:** Добавить в [tz_architecture.md](tz_architecture.md) раздел «Спектр автономности» с:
- Таблицей уровней (0-4) для каждого инструмента/агента
- Паттерном Human-in-the-loop в PHP (очередь утверждений + UI)
- Конфигурацией порогов доверия в `project.php`
- Метриками для принятия решения о повышении уровня

### 4. Масштабирование — «Выдержит рост?»

**Ключевой пробел.** Масштабирование затрагивается косвенно (3 отдельных Django-сервиса), но без целенаправленной стратегии.

**Что уже есть / запланировано:**

| Решение | Где описано | Статус |
|---------|-----------|--------|
| Разделение Django на 3 сервиса (core, agents, mas) | [tz_architecture.md](tz_architecture.md) | Планируется (Этап 4-5) |
| Rate limiting на PHP (`/api/send`: 10 req/60 sec) | MEMORY.md | Реализовано |
| Отдельные Dockerfile для каждого Django-сервиса | [tz_architecture.md](tz_architecture.md) | Планируется |

**Чего не хватает:**

| Пробел | Влияние | Приоритет |
|--------|---------|-----------|
| Стратегия горизонтального масштабирования | Один экземпляр = потолок нагрузки | Высокий (Этап 4) |
| Docker Compose / Kubernetes конфигурация | Ручное развёртывание = ошибки | Высокий (Этап 4) |
| Rate limiting между сервисами | Django Agents может быть перегружен | Средний |
| Кэширование (Redis/Memcached) | Повторные запросы = лишняя нагрузка | Средний |
| Очередь задач (Celery/RQ) | AI-задачи блокируют HTTP-workers | Высокий (Этап 5) |
| Health checks и автоматический рестарт | Упавший сервис = ручное восстановление | Высокий |
| Метрики нагрузки (Prometheus/Grafana) | Нет данных для решений о масштабировании | Средний |

**Рекомендация:** Создать отдельный ТЗ `tz_scaling.md` или добавить раздел в [tz_architecture.md](tz_architecture.md):
- `docker-compose.yml` с replicas, health checks, resource limits
- Очередь задач для AI-агентов (Celery + Redis)
- Кэширование справочников в PHP (file-based или Redis)
- Метрики и алерты

### 5. Универсальность — «Работает в любом контексте?»

**Что уже есть / запланировано:**

| Решение | Где описано | Статус |
|---------|-----------|--------|
| MCP Server — стандартный интерфейс для любой AI-модели | [tz_protocols.md](tz_protocols.md) | Планируется (Этап 4-5) |
| MCP Tools/Resources/Prompts — три универсальных примитива | [tz_protocols.md](tz_protocols.md) | Планируется |
| OpenAPI-спецификации Django-сервисов | [tz_protocols.md](tz_protocols.md) | Планируется |
| Единый JSON-формат логов для всех компонентов | [tz_architecture.md](tz_architecture.md) | Планируется |

**Чего не хватает:**

**Универсальный контракт задачи** — единый формат для ЛЮБОГО агента, ЛЮБОГО типа задачи:

```json
{
  "task_id": "uuid",
  "task_type": "string",
  "priority": "low|normal|high",
  "input": {
    "type": "string",
    "data": {}
  },
  "constraints": {
    "timeout_ms": 60000,
    "max_retries": 3,
    "confidence_threshold": 0.8
  },
  "context": {
    "tenant_id": "string",
    "trace_id": "string",
    "user_id": "string",
    "source": "php|n8n|mas|mcp"
  }
}
```

Сейчас `AgentInput` содержит `payload: dict[str, Any]` — это «мешок», не контракт. Каждый агент интерпретирует payload по-своему.

**Не хватает:**
- Типизированного `input.type` (image, text, structured_data, url)
- Единой системы constraints (timeout, retries, confidence threshold)
- Стандартного `context` с метаданными источника
- Единого формата ошибок между PHP, n8n, Django

**Рекомендация:** Добавить в [tz_protocols.md](tz_protocols.md) раздел «Универсальный контракт задачи» — формализовать структуру `TaskRequest`/`TaskResponse`, используемую на всех уровнях системы.

## Парадоксы и компромиссы

Оптимизация по одной оси неизбежно создаёт трение на другой. Понимание этих парадоксов помогает принимать осознанные архитектурные решения.

### 1. Переиспользуемость vs Гибкость

```
Переиспользуемость            Гибкость
«Стандартный интерфейс»       «Каждый случай уникален»
         ↑                          ↑
         └────── КОНФЛИКТ ──────────┘

Абстракция BaseAgent → единый pipeline,
но каждый новый агент может нуждаться
в другом pipeline (streaming, batch, real-time)
```

**Решение в системе:** `BaseAgent` обеспечивает базовый pipeline (validate → process → return), но разрешает переопределение `run()`. MAS Orchestrator поддерживает стратегии (`strategies/`), позволяя менять координацию без изменения агентов.

### 2. Тиражируемость vs Универсальность

```
Тиражируемость                Универсальность
«Стандартный деплой»          «Любой контекст»
         ↑                          ↑
         └────── КОНФЛИКТ ──────────┘

Профиль small-business предполагает PHP + n8n,
но универсальный заказчик может использовать
другой orchestrator (Temporal, Airflow) вместо n8n
```

**Решение в системе:** Профили (`small-business`, `enterprise`, `saas`) покрывают типовые сценарии. Интерфейсы (`AutomationGatewayInterface`) позволяют заменить n8n на другой оркестратор, но scaffold и документация заточены под n8n. Компромисс: стандартизация на n8n как основном оркестраторе, с возможностью замены через DI.

### 3. Автономность vs Гибкость

```
Автономность                  Гибкость
«Система решает сама»         «Легко изменить поведение»
         ↑                          ↑
         └────── КОНФЛИКТ ──────────┘

AI-агент принимает решения на основе обученной модели.
Изменить его поведение = переобучение, не конфиг.
```

**Решение в системе:** MCP Prompts — инструкции, которые можно менять без переобучения модели. Стратегии MAS Orchestrator — конфигурируемые pipeline'ы. Пороги доверия (confidence threshold) в `project.php` — регулировка «жёсткости» автономности через конфиг.

### 4. Масштабирование vs Гибкость

```
Масштабирование               Гибкость
«Предсказуемая нагрузка»      «Динамичная структура»
         ↑                          ↑
         └────── КОНФЛИКТ ──────────┘

Горизонтальное масштабирование требует stateless-сервисов.
Гибкая конфигурация может требовать shared state.
```

**Решение в системе:** PHP-сайт уже stateless (JSON-контент, без БД). Django-сервисы проектируются stateless (состояние в БД, кэш в Redis). Конфигурация (`project.php`, `.env`) — статическая, загружается при старте.

|---|:---:|:---:|:---:|:---:|:---:|:---:|
| **Переисп.** | — | + | ~ | ~ | + | - |
| **Тираж.** | + | — | ~ | ~ | - | ~ |
